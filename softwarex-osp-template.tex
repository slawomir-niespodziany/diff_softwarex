Embedded systems are nowadays getting bigger and more complex as the technological advancements allow them to consist of more and more functionalities. Each system often comes in several variants, having customization options, subscription plans and update capabilities. Software is a vital part of an embedded system and grows accordingly to the system itself. While in the past it was sufficient to bootstrap and maintain a simple project for an embeddeded device, this is no longer the solution. It requires an up--front plan for configuration, turning functionalities on and off, adding new features not only at production, but also during the devices lifetime or simply fixing unavoidable bugs released unintentionally. The number of possible variants does not allow for cloning the project and deploying it with modifications each time. Such approach almost immediately breaks maintainability constraints. Another approach would be to keep all the possible variants within a single project, but this fall short in the context of complexity, build--time, IP--management, CI/CD compatibility. The bigger the system, the more problemmatic customization of its software becomes.

Diff - \textit{Dependency Injection Framework} - \textit{First} variant - is a generic solution for embedded systems implemented using C++14 (and higher standards). It is a framework for implementing modular embedded applications with reusability and maintainability in mind from day one. It is compatibile with tools and methodologies widely used in embedded development. Its origin is nothing new. It was inspired by the mechanisms used in modern, scalable systems like the Spring framework. The basic principle is to use Json file with information controlling the structure and configuration of the resulting application. This data--driven approach mimicks large--scale systems whose functionality is often defined via metadata files e.g. IaC, CI/CD. The dependency injection pattern highly utilized at the code side allows the framework engine to handle lifetime and dependencies between application building--blocks, called \textit{components}. It allows for their loose coupling, which is a critical requirement for the codebase to be maintainable and scalable.

\subsection{Modular architecture}
Diff imposes certain design decisions on the architecture of the implemented software. Their point is to make software modular. However, this is only a low level (physical) constraint, allowing the user to decide on the logical structure. The application is composed of components of two kinds. Firstly, \textit{interface components} are lightweight and define contracts. Secondly, more heavyweight, regular components, which provide implementations are simply called \textit{components}. \textit{Interface components} are provided in the form of abstract classes. They consis of API definitions, which dictate how \textit{components} interact with each other.
The \textit{components} provide implementations of those contracts. While they are similarily implemented in form of C++ classes, they consist of logic and are instantiable. \textit{Components} accept interface components as their contructor arguments, thus realizing the dependency injection pattern, allowing for inter--component interactions.
The user provides implementations of both component types in form of header--only libraries (for \textit{interface components}) or statically--linked or shared libraries for \textit{components}. Their number, granularity and functionality depends on the particular project requirements. Components in their physical, library, form are referred to as \textit{modules}. The final application binary is linked with a subset of provided \textit{modules}.

\subsection{Loose coupling}
While the architecture at the lowest level is heavily imposed by the framework, the benefits resulting from following the proposal are as follows. While some of the listed arguments may seem very obvious in other software devlopment areas, they are not necessarily industry--standards in embedded software engineering.

\subsubsection{Validation}
Component tests are implemented and run in complete isolation from other components. Common unit test frameworks can be utilized for this purpose (e.g. GTest). Loose coupling allows for implementing most of the components using pure C++ - without the dependence on platform and hardware. This opens the possibility to run and validate implementations firstly - while development, on host machines (e.g. x86 architecture) and secondly, while deployment - within CI/CD pipelines, using containers.

\subsubsection{Deployment}
Each module is built at the time of its deployment, e.g. when it is submitted. The source code is immediately compiled and tested. Modules are built in form of libraries for all the platforms up--front (e.g. x86, ARM). From the same source code, a building block becomes available for any environment which may be necessary. This can be automated at the CI/CD level. The modules are stored in artifact repository (e.g. Artifactory).

This is useful for components intended for usage in multiple environments. A good example may be a communication protocol layer, utilized in both host and device environments, which differ significantly (different architecture, compiler, resources). Havin ig in form of a library, which may be optimized for footprint on one architecture and for performance on the other allows for simply using an existing component in the following steps, rather that figureing how to rebuild (or worse - rewrite) an IP for another environment.

This also reduces the build time of the resulting binary. As the modules are already compiled, available aout--of--the--box. The only step left is to link them with the runnable, or even simpler - upload to the device, if the module is shared library.

\subsubsection{CI/CD compatibility}
Because every \textit{component} can be build independently from the others (apart from the lightweight \textit{interface components}), any modern CI/CD pipeline can process it using standard tools. Toolchain for any target platform can be used within a dedicated container, used for the build process. Multiple containers provide the capability to compile for multiple platforms. The process can thus be paralellized in cloud environment.

\subsubsection{Versioning and multi--platform ability}
The intrinsic compatibility with modern package management tools, such as Conan 2.0, allows for their usage for versioning and deploying modules for multiple platforms. Both \textit{interface components} and regular \textit{components} require versioning. While the first ones are deployed in form of header--only libraries, which makes them natively multi--pltform, the regular \textit{components} need to be build for all the intended pltforms separately. This can be easily automated with such tools and contenerization, where each container consists of a toolchain and configuration for a single platform and the source project is platform independent. Building the project within a selected container produces a binary for the corresponding platform.

% where the compiler and other platform--specific tools are within a container, the container can be used for offline builds and ci/cd based builds as usually is the case for cicd environments

\subsubsection{Artifact management}
Having multiple component versions for multiple platforms requires some organized means for storing the corresponding artifacts. One of the available tools is the Artifactory repository. It manages the created artifacts and makes them available for further usage. It is of crucial importance, that the selected package management tool shall easily integrate with it, to allows for smooth dependency usage. For the Artifactory and Conan 2.0 this is true. It is particularily important that the artifact repository shall be a non--local resource. Firstly, for consistency - all the involved devlopers shall use the same repository. Secondly, and most importantly, the CI/CD pipeline shall perform component build and deployment. As this will most likely happen within some cloud environment, the artifact repo shall be accessible from there too.

\subsubsection{Binary composition}
The resulting binary shall be composed of multiple components. The build process must be able to pull and include the required ones. This is partially handled by the already mentioned tools. However, they only operate at the level of a module, while composing an executable for the given topology also requires the knowledge about the interactions between components - their dependencies, implemented interfaces configuration capabilities. This infor is provided by the integrator of the runnable, but for two reasons an additional tool is required. Firstly, the Json file provided shall be validated as early as possible, or the binary may have problems at startup or may not build at all. Secondly, for the integrators covenience the list of available and compatibile components and their versions shall be aggregated in one place. From there a convenient method to compose the desired topology shall also be provided. This is realized through a custom desktop application \textit{diff-forge}.

It is worth emphasizing, that nowadays many embedded projects are still executed as they were 20 years ago. This resembles using a single SVN repository, with separate branches for multiple software variants, built manually using an IDE in Windows--based developers PC. This is one end of the scale and there are obviously cases in the whole spectra - from the disfunctional ones (as in this example) up to the well managed, adapting the newest trends and methodologies. The point is, however, that there are little industry standards, compared to e.g. webapp development, where typical problems are solved at higher levels, allowing the work to focus on application specific issues.

\subsection{Physical and logical structure}
Diff imposes physical structure - not for the application, but rather the environment, where it is developed and deployed. Nevertheless, this has some transitive impact on the internal applications architecture. From this perspective the app is composed of multiple separate components - which is in fact the main point. From there the app can be further architected into component groups. This may resemble layered architecture, hexagonal, or any of the designers choice. One of the possibilities, especially for embedded systems, where certain level of variablity is required, is as follows - the design is split into three layers:
\begin{itemize}
    \item applicaion
    \item system
    \item platform
\end{itemize}

% emphasize this is only an example of app architecture, though well argumented.

Further sections explain this using the example of permanent storage requirement.

\subsubsection{Application layer}
This is a layer where the application logic resides. It is mainly driven by the business requirements, not necessarily depending on the used MCU/SoC, or the overall ssytem architecture. It may obviously assume certain restrictions e.g. on the underlying hardware, but does not define it specifically.

The app may require some way to store data permanently, it may define the size fo required sotrage, its non--functional parameters, like access time, durability, but does not define anything specific.

\subsubsection{System layer}
This layer fulfills the requirements imposed by the application layer, providing certain resources to it. The difference is however, that the top layer only soecifies the requirement for certain use--cases and this layer specifies how they are provided. This is crucial for system flexibility.

This layer defines what is the target device providing the required functionality. It may be a flash--memory communicating over SPI bus if it fulfills the requiremetns falling down from the upper layer.

\subsubsection{Platform layer}
This specifies how the software integrates with the platform it is running within. This is crucial for software portability.

This layer provides the integration method of the driven device with the underlying platform. E.g. it provides the SPI driver to allow for communicating with the selected SPI--flash.

\subsubsection{Flexibility provided}
Now, if the memory chip unavailable (out--of--stock, or not produced anymore), the design may incorporate another similar chip. Then the only place where the software requires an update is the System layer - the module responsible for handling the specific chip can be replaced. Then if two software variants are deployed they only differ in that specific component.

If the replaced part is the platform the software runs on (e.g. the CPU/MCU), the only plce where the design requires changes is the Platform layer - in this case particularily the SPI driver.

Diff is a header--only library imported into users project, in particular it is utilized by every \textit{component}. It provides a set of template classes used for creating components, allowing them to be instantiated and managing the created topology. It allows to access the instantiated components from outside the created topology.

\subsection{Component implementation}
A \textit{component} is implemented in form of a C++ class. To make the framework aware of it, it derives from a CRTP template \textit{diff::Component<>} provided by the diff library. A component may both depend and be a dependency for other components.

\subsubsection{Consuming a dependency}
To utilize functionalities of other components, a common contract - \textit{interface component} - firstly needs to be defined in form of an abstract C++ class. It defines the required functionality, but does not specify the implementation. A \textit{component} consumes a dependency through its constructor (Figure~\ref{fig:component_implementation}). It accepts a reference to an abstract interface class and uses its functionality utilizing polymorphism provided by C++.

\subsubsection{Exposing a dependency}
To expose its functionality, the component derives from \textit{interface components} implemented as abstract classes. Usual inheritance, with no side--effects, is still available. To make the interface exposed, it shall be designated with $as<>$ trait and declared within the $diff::Component<>$ template. This creates the inheritance relation and notifies the framework about the exposed interface. The following example depicts the possibilities:

\begin{figure}[ht]
    \centering
    {\scriptsize\begin{verbatim}
class MyComponent : 
    public diff:Component<MyComponent, 
                          diff::as<ExposedInterface1>
                          diff::as<ExposedInterface2>>, 
    public OtherClass {
public:
    MyComponent(ConsumedInterface & firstDependency) {
        ...
    }
};
\end{verbatim}}
    \caption{Component visible for the framework uses a CRTP template class $Component<>$. Interfaces through which the component is exposed as a dependency shall be abstract classes and are declared within that template with $as<>$ traits. Usual inheritance is also possible - the framework is not aware of it. In this example the component is exposed through $ExposedInterface1$ and $ExposedInterface2$ interfaces, while it simply inherits after the $OtherClass$ - without any side--effects. TODO CONSTRUCTOR}
    \label{fig:component_implementation}
\end{figure}

An instance created by the framework always has a unique identifier of type std::string assigned by the user. When the component exposes interfaces they become available as dependencies for other components and are too identified by the same id. This is possible because an interface can only be inherited once by a C++ class. While this makes dependency identification simple, it has a serious disadvantage. A component may not expose multiple interfaces of the same type this way. For this reason diff provides a method for exposing multiple internal objects as dependencies. This is realized through a \textit{side} member function of the following signature. It is intended to return references to the internal objects and id suffixes for each. The resulting id for each such dependency is composed with the owning instance id and each suffix concatenated with an underscore character.

\begin{figure}[ht]
    \centering
    {\scriptsize\begin{verbatim}
using namespace diff;

class MyComponent : 
    public Component<MyComponent, ..., side<MyInterface>> {
public:
    ... 

    void side(std::map<std::string, std::reference_wrapper<MyInterface>> &sideDependencies) {
        ...
    }
};
\end{verbatim}}
    \caption{Component visible for the framework uses a CRTP template class $Component<>$. Interfaces through which the component is exposed as a dependency shall be abstract classes and are declared within that template with $as<>$ traits. Usual inheritance is also possible - the framework is not aware of it. In this example the component is exposed through $MyFirstInterface$ and $MySecondInterface$ interfaces, while it simply inherits after the $OtherClass$ - without any side--effects.}

\end{figure}

\subsection{Wraping in a module}




A \textit{component} may provide implementations for arbitrary number of \textit{interface components}. Thus it can be then injected as a dependency of one of the implemented interface types. It implements these interfaces either directly or exposes internal objects which implement them. The framework must obtain their references to be able to inject them as dependencies for other components. For this reason a \textit{component} derives from a CRTP template class \textit{Component<>}, which informs the framework about the interfaces it implements directly and indirectly. The tempalte also then defines a factory for the component. Traits \textit{as<>} and \textit{side<>} are used to designate interfaces implemented directly and indirectly. Indirectly implemented interfaces are referenced by the framework through a member function of the following signature:

\begin{verbatim}
void Component::side(std::map<std::string, std::reference_wrapper<Interface>> & sideDependencies);
\end{verbatim}

An exemplary component differs a bit, compared to a pure class implemented out of the context of diff. However the impact is not significant and can if required - a mock can be introduced to make the framework behavior a \textit{no--op}.
