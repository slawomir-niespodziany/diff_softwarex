Embedded systems are nowadays getting bigger and more complex as the technological advancements allow them to consist of more and more functionalities. Each system often comes in several variants, having customization options, subscription plans and update capabilities. Software is a vital part of an embedded system and grows accordingly to the system itself. While in the past it was sufficient to bootstrap and maintain a simple project for an embeddeded device, this is no longer the solution. It requires an up--front plan for configuration, turning functionalities on and off, adding new features not only at production, but also during the devices lifetime or simply fixing unavoidable bugs released unintentionally. The number of possible variants does not allow for cloning the project and deploying it with modifications each time. Such approach almost immediately breaks maintainability constraints. Another approach would be to keep all the possible variants within a single project, but this fall short in the context of complexity, build--time, IP--management, CI/CD compatibility. The bigger the system, the more problemmatic customization of its software becomes.

Diff - \textit{Dependency Injection Framework} - \textit{First} variant - is a generic solution for embedded systems implemented using C++14 (and higher standards). It is a framework for implementing modular embedded applications with reusability and maintainability in mind from day one. It is compatibile with tools and methodologies widely used in embedded development. Its origin is nothing new. It was inspired by the mechanisms used in modern, scalable systems like the Spring framework. The basic principle is to use Json file with information controlling the structure and configuration of the resulting application. This data--driven approach mimicks large--scale systems whose functionality is often defined via metadata files e.g. IaC, CI/CD. The dependency injection pattern highly utilized at the code side allows the framework engine to handle lifetime and dependencies between application building--blocks, called \textit{components}. It allows for their loose coupling, which is a critical requirement for the codebase to be maintainable and scalable.

\subsection{Modular architecture}
Diff imposes certain design decisions on the architecture of the implemented software. Their point is to make software modular. However, this is only a low level (physical) constraint, allowing the user to decide on the logical structure. The application is composed of components of two kinds. Firstly, \textit{interface components} are lightweight and define contracts. Secondly, more heavyweight, regular components, which provide implementations are simply called \textit{components}. \textit{Interface components} are provided in the form of abstract classes. They consis of API definitions, which dictate how \textit{components} interact with each other.
The \textit{components} provide implementations of those contracts. While they are similarily implemented in form of C++ classes, they consist of logic and are instantiable. \textit{Components} accept interface components as their contructor arguments, thus realizing the dependency injection pattern, allowing for inter--component interactions.
The user provides implementations of both component types in form of header--only libraries (for \textit{interface components}) or statically--linked or shared libraries for \textit{components}. Their number, granularity and functionality depends on the particular project requirements. Components in their physical, library, form are referred to as \textit{modules}. The final application binary is linked with a subset of provided \textit{modules}.

\subsection{Loose coupling}
While the architecture at the lowest level is heavily imposed by the framework, the benefits resulting from following the proposal are as follows. While some of the listed arguments may seem very obvious in other software devlopment areas, they are not necessarily industry--standards in embedded software engineering.

\subsubsection{Validation}
Component tests are implemented and run in complete isolation from other components. Common unit test frameworks can be utilized for this purpose (e.g. GTest). Loose coupling allows for implementing most of the components using pure C++ - without the dependence on platform and hardware. This opens the possibility to run and validate implementations firstly - while development, on host machines (e.g. x86 architecture) and secondly, while deployment - within CI/CD pipelines, using containers.

\subsubsection{Deployment}
Each module is built at the time of its deployment, e.g. when it is submitted. The source code is immediately compiled and tested. Modules are built in form of libraries for all the platforms up--front (e.g. x86, ARM). From the same source code, a building block becomes available for any environment which may be necessary. This can be automated at the CI/CD level. The modules are stored in artifact repository (e.g. Artifactory).

This is useful for components intended for usage in multiple environments. A good example may be a communication protocol layer, utilized in both host and device environments, which differ significantly (different architecture, compiler, resources). Havin ig in form of a library, which may be optimized for footprint on one architecture and for performance on the other allows for simply using an existing component in the following steps, rather that figureing how to rebuild (or worse - rewrite) an IP for another environment.

This also reduces the build time of the resulting binary. As the modules are already compiled, available aout--of--the--box. The only step left is to link them with the runnable, or even simpler - upload to the device, if the module is shared library.

\subsubsection{CI/CD compatibility}
Because every \textit{component} can be build independently from the others (apart from the lightweight \textit{interface components}), any modern CI/CD pipeline can process it using standard tools. Toolchain for any target platform can be used within a dedicated container, used for the build process. Multiple containers provide the capability to compile for multiple platforms. The process can thus be paralellized in cloud environment.
